# 第6章:AIテストとセキュリティ

## 講座概要

> "Software errors can dash user trust in a product/company and incur huge financial costs"

LLMが大部分のコードを記述する場合、エラーを防ぐために広範なガードレールが必要です。AI時代のソフトウェア品質は、これまで以上に重要です。

## 2. 脆弱性検出技術

### SAST(静的アプリケーションセキュリティテスト)
- **タイプ**: White box testing(ホワイトボックステスト)
- **分析**: バイナリファイルとソースコード
- **段階**: ソフトウェア開発ライフサイクルの早期(コストが最も低い)
- **技術**: コードベースのパターンマッチングスキャン
- **検出**: SQLインジェクション、コマンドインジェクション、クロスサイトスクリプティング

### DAST(動的アプリケーションセキュリティテスト)
- **タイプ**: Black box testing(ブラックボックステスト)
- **方法**: 現実世界のハッカーの行動をシミュレートして脆弱性を発見
- **段階**: SDLC全体、誤検出が少ない
- **技術**:
  - Input fuzzing(入力ファジング)
  - Manipulating session tokens(セッショントークン操作)
  - Configuration/header testing(設定/ヘッダーテスト)
  - Brute force rate-limit tests(ブルートフォースレート制限テスト)

### SCA(ソフトウェア構成分析)
- **分析**: アプリケーションが使用するオープンソースソフトウェアパッケージの詳細分析
- **方法**: パッケージマネージャー、Infrastructure as Code、pullイメージを分析
- **技術**: 依存関係の脆弱性をスキャン

### テスト手法の比較

| 手法 | タイプ | 段階 | 利点 |
|------|------|------|------|
| SAST | White box | 早期 | コストが低く、カバレッジが広い |
| DAST | Black box | SDLC全体 | 誤検出が少なく、ランタイムの問題を発見 |
| SCA | 依存関係分析 | ビルド時 | サードパーティライブラリの脆弱性をチェック |

## 4. AI Agentの新型攻撃ベクトル

| 攻撃タイプ | 説明 |
|----------|------|
| **Prompt Injection** | 隠されたまたは誤解を招く指示によってAIを意図した動作から逸脱させる |
| **Tool Misuse** | 欺瞞的なプロンプトでagentを操作し、統合ツールを悪用させる |
| **Intent Breaking** | agentの計画を操作し、元の意図から遠ざかるようアクションをリダイレクトする |
| **Identity Spoofing** | 侵害された認証を利用して正当なagentになりすます |
| **Code Attacks** | agentのコード実行能力を悪用して実行環境への不正アクセスを取得する |

---

## 5. セキュリティテストにおけるLLMの限界

| 限界 | 説明 |
|--------|------|
| **高い誤検出率** | Claude Code/Codexの誤検出率は50-100%、脆弱性の種類に依存 |
| **非決定論的分析** | 同じプロンプトを複数回実行すると異なる結果が得られる。すべての脆弱性を発見したかどうかをどう判断するか? |
| **Context Rot** | すべてのコンテキストが同等に有効というわけではない |
| **Compaction** | 要約によってコンテキストが圧縮され、情報が失われる |
| **非現実的なベンチマーク** | 既存のベンチマークはしばしば非現実的であり、LLMの評価が困難 |

### 従来のSASTとの比較
- AI SASTの誤検出率:50-100%(脆弱性に依存)
- 従来のSASTの誤検出率:50%+
