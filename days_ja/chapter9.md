> "Monitoring software systems in production remains a mission-critical task"

## 1. 従来のSRE職責

| 職責 | 説明 |
|------|------|
| **運用監視** | オンコール、トラブルシューティング、インフラストラクチャ管理とセキュリティ |
| **インシデント解決** | 複数の異なるソースとチームから情報を収集・統合 |
| **手順書の保守** | 通常古くなっているトラブルシューティング手順書の維持 |
| **燃え尽き症候群** | オンコールのローテーションによるSREの疲弊 |

### 現代化の課題
- クラウドネイティブアーキテクチャへの移行
- コンテナ化されたワークロードとKubernetes
- より多くのデータ、依存関係、システムの複雑性の導入

## 3. インシデント対応手順書

### シナリオ
> 午前3時12分、PagerDutyからの通知を受け取りました。データベースクエリで500エラーのスパイクが発生しています。どうしますか?

### 対応ステップ

```
1. Acknowledge and assess(確認と評価)
2. Check DB + app(データベースとアプリケーションのチェック)
3. Identify recent changes(最近の変更の特定)
4. Localize blast radius(影響範囲の特定)
5. Execute mitigations(緩和策の実行)
6. Stabilize + monitor(安定化+監視)
7. Communicate(コミュニケーション)
8. Document(ドキュメント化)
```

### 主要指標

| 指標 | 説明 |
|------|------|
| **MTTR** | Mean-time-to-repair 平均修復時間 |
| **On-call 人数** | インシデントに対応するエンジニアの数 |
| **SLA** | 顧客に報告されるサービスレベルアグリーメント |

## 5. AI SREの特徴

> "AI SRE" = Agentic system across observability stack and clouds

| 特徴 | 説明 |
|------|------|
| **動的ナレッジグラフマッピング** | システムコンポーネント間の関係をリアルタイムで理解 |
| **可観測性スタックとクラウド全体のエージェントシステム** | クロスプラットフォームのインテリジェント監視 |
| **リアルタイムナラティブ** | 発生している状況のリアルタイムな説明を生成 |
| **根本原因分析** | 可能性のある根本原因と裏付け証拠の特定 |
| **修復提案** | 明確な修復手順の推奨 |
| **説明可能性と監査可能性** | 予測/推論の説明可能性と監査可能性を重視 |

---

## 6. AIは何を変えたのか?

### 知識のスケール化
AIは組織とサービスの知識を拡張することを約束します:
- 情報はもはや、文書化されていない依存関係、脆弱なレガシーサービス、高リスクなインシデントでのみ表面化する癖を唯一知っているエンジニアに限定されません

### 開発プロセスの変化
- 自動化によりレビュー時間を短縮し、早期に問題を発見
- 開発者はAIの提案を通じてベストプラクティスを学習
- AIはすべてのコードレビューで同じ基準を適用
- AIが日常的なチェックを処理し、人間は複雑なロジックに集中
- AIシステムは時間の経過とデータの蓄積により改善
- 最新のAIコードレビューツールは、より深いコンテキスト分析とパターン認識を提供

---

## 7. AI SREの制限

| 制限 | 説明 |
|------|------|
| **インシデントの複雑性** | 解決できるインシデントの複雑度に制限がある |
| **本番スタックの異質性** | 最新の本番スタックの多様性 |
| **コード修復能力** | 検出された内容に基づいて実際のコードを修正する能力 |
| **監視のガーデニング** | 優れた根本原因分析には優れた監視の維持が必要 |
| **セキュリティリスク** | 新たな攻撃ベクトルになる可能性 |

### 最終目標
- すべてのプロバイダーは根本原因分析から始める
- 最終目標はコードの修正
